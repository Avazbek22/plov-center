## User Scenarios & Testing

### User Story 1 — Навигация админ-панели (Priority: P1)

Администратор видит боковое меню с пунктами: Дашборд, Категории, Блюда, Контент. Может переключаться между разделами. Текущий раздел подсвечен.

**Why this priority**: Без навигации невозможно переключаться между разделами. Необходимо для работы всех остальных сторис.

**Independent Test**: Войти → убедиться, что боковое меню видно → кликнуть на каждый пункт → убедиться, что переход работает и текущий пункт подсвечен.

**Acceptance Scenarios**:

1. **Given** администратор авторизован, **When** он на любой странице админки, **Then** видно боковое меню с пунктами навигации
2. **Given** боковое меню, **When** администратор кликает на "Категории", **Then** открывается раздел категорий и пункт "Категории" подсвечен
3. **Given** боковое меню, **When** администратор кликает на "Блюда", **Then** открывается раздел блюд и пункт подсвечен

---

### User Story 2 — Управление категориями (Priority: P1)

Администратор открывает раздел "Категории" в админ-панели, видит список всех категорий с количеством блюд. Может создать новую категорию, отредактировать название и порядок сортировки, переключить видимость, изменить порядок сортировки, удалить пустую категорию.

**Why this priority**: Категории — основа меню. Без них нельзя создавать блюда. Это первый шаг для наполнения сайта контентом.

**Independent Test**: Войти → "Категории" → создать категорию "Супы" → убедиться, что она появилась в списке → отредактировать название на "Первые блюда" → переключить видимость → удалить.

**Acceptance Scenarios**:

1. **Given** администратор авторизован, **When** он переходит в раздел "Категории", **Then** отображается таблица категорий с колонками: название, порядок, видимость, кол-во блюд, действия
2. **Given** администратор нажимает "Добавить категорию", **When** вводит название и нажимает "Сохранить", **Then** категория появляется в списке
3. **Given** категория существует, **When** администратор нажимает "Редактировать", меняет название и сохраняет, **Then** название обновляется в списке
4. **Given** категория существует, **When** администратор нажимает переключатель видимости, **Then** видимость категории меняется
5. **Given** категория не содержит блюд, **When** администратор нажимает "Удалить" и подтверждает, **Then** категория удаляется из списка
6. **Given** категория содержит блюда, **When** администратор нажимает "Удалить", **Then** система показывает ошибку, что категория содержит блюда

---

### User Story 3 — Управление блюдами (Priority: P1)

Администратор открывает раздел "Блюда", видит список всех блюд с фильтрацией по категории. Может создать блюдо с названием, описанием, ценой, фото и категорией. Может отредактировать любое поле, переключить видимость, удалить блюдо.

**Why this priority**: Блюда — основной контент сайта. Без управления блюдами сайт бесполезен. Равный приоритет с категориями, т.к. зависит от них.

**Independent Test**: Войти → "Блюда" → создать блюдо "Плов" с ценой 45000 → загрузить фото → убедиться, что блюдо появилось → отредактировать цену → переключить видимость → удалить.

**Acceptance Scenarios**:

1. **Given** администратор в разделе "Блюда", **When** страница загружается, **Then** отображается таблица блюд с колонками: фото, название, категория, цена, видимость, действия
2. **Given** список блюд отображён, **When** администратор выбирает категорию в фильтре, **Then** отображаются только блюда этой категории
3. **Given** администратор нажимает "Добавить блюдо", **When** заполняет форму (название, категория, цена обязательны; описание, фото опционально) и сохраняет, **Then** блюдо появляется в списке
4. **Given** форма создания блюда, **When** администратор загружает фото, **Then** фото загружается на сервер и отображается в превью
5. **Given** блюдо существует, **When** администратор редактирует и сохраняет, **Then** изменения отображаются в списке
6. **Given** блюдо существует, **When** администратор переключает видимость, **Then** статус видимости меняется
7. **Given** блюдо существует, **When** администратор удаляет и подтверждает, **Then** блюдо исчезает из списка

---

### User Story 4 — Управление контентом сайта (Priority: P2)

Администратор открывает раздел "Контент сайта" и видит две секции: "О нас" и "Контакты". Может отредактировать текст "О нас" с фотографией, а также контактную информацию: адрес, телефон, часы работы, код карты.

**Why this priority**: Контент сайта важен, но меняется редко. Менее критичен, чем управление меню.

**Independent Test**: Войти → "Контент" → изменить текст "О нас" → загрузить фото → обновить адрес в контактах → убедиться, что всё сохранилось.

**Acceptance Scenarios**:

1. **Given** администратор в разделе "Контент", **When** страница загружается, **Then** отображаются формы для "О нас" и "Контакты" с текущими значениями
2. **Given** секция "О нас", **When** администратор меняет текст и/или загружает фото и нажимает "Сохранить", **Then** контент обновляется
3. **Given** секция "Контакты", **When** администратор меняет адрес/телефон/часы/карту и нажимает "Сохранить", **Then** контактная информация обновляется
4. **Given** администратор сохраняет контент, **When** обновление успешно, **Then** отображается уведомление об успехе

---

### Edge Cases

- Что происходит при попытке удалить категорию с блюдами? → Сервер возвращает ошибку, UI показывает сообщение
- Что если загружаемое фото слишком большое? → Сервер возвращает ошибку валидации, UI показывает сообщение
- Что если два администратора одновременно редактируют одну категорию? → Последний сохранивший перезаписывает
- Что если при сохранении формы сервер недоступен? → UI показывает сообщение об ошибке сети
- Что если название категории уже существует? → Сервер возвращает конфликт, UI показывает сообщение

## Requirements

### Functional Requirements

- **FR-001**: Система ДОЛЖНА отображать список категорий с возможностью создания, редактирования, удаления и переключения видимости
- **FR-002**: Система ДОЛЖНА отображать список блюд с фильтрацией по категории и возможностью CRUD-операций
- **FR-003**: Система ДОЛЖНА позволять загрузку изображений для блюд и секции "О нас"
- **FR-004**: Система ДОЛЖНА отображать формы редактирования контента сайта (О нас, Контакты)
- **FR-005**: Система ДОЛЖНА отображать боковое меню навигации по разделам админки
- **FR-006**: Система ДОЛЖНА показывать подтверждение перед удалением любой сущности
- **FR-007**: Система ДОЛЖНА отображать серверные ошибки пользователю в понятном виде
- **FR-008**: Система ДОЛЖНА валидировать обязательные поля на клиенте до отправки на сервер
- **FR-009**: Система ДОЛЖНА показывать уведомления об успешных операциях (создание, обновление, удаление)
- **FR-010**: Система ДОЛЖНА поддерживать изменение порядка сортировки категорий
- **FR-011**: Система ДОЛЖНА отображать превью загруженного изображения перед сохранением

### Key Entities

- **Категория (Category)**: Группировка блюд. Имеет название, порядок сортировки, видимость. Связана с блюдами (один-ко-многим)
- **Блюдо (Dish)**: Позиция меню. Имеет название, описание, цену, фото, категорию, порядок сортировки, видимость
- **Контент сайта (SiteContent)**: Контент "О нас" (текст + фото) и контактная информация (адрес, телефон, часы, карта)

## Success Criteria

### Measurable Outcomes

- **SC-001**: Администратор может создать категорию и блюдо за менее чем 1 минуту
- **SC-002**: Все CRUD-операции завершаются в течение 3 секунд для пользователя
- **SC-003**: 100% серверных ошибок валидации отображаются пользователю с понятным сообщением
- **SC-004**: Навигация между разделами происходит без перезагрузки страницы
- **SC-005**: Загрузка изображения с превью работает корректно для файлов до 5 МБ

## Assumptions

- Используется существующая авторизация из фичи 001 (JWT + AuthProvider)
- Бэкенд полностью реализован — все API-эндпоинты уже работают
- Один администратор — нет необходимости в управлении пользователями
- Формат ответов API соответствует существующим контрактам бэкенда
- Загрузка файлов через multipart/form-data на существующий эндпоинт uploads
- Порядок сортировки управляется числовым полем SortOrder (не drag-and-drop в MVP)

## Scope

### В scope

- CRUD-интерфейс для категорий
- CRUD-интерфейс для блюд с загрузкой фото
- Формы редактирования контента сайта (О нас + Контакты)
- Боковая навигация админки
- Валидация форм на клиенте
- Отображение серверных ошибок
- Уведомления об успешных операциях

### Вне scope

- Управление администраторами (нет API)
- Drag-and-drop сортировка (используем числовое поле в MVP)
- Пагинация (объём данных кафе не требует)
- Поиск по блюдам/категориям
- Массовые операции (bulk delete, bulk visibility toggle)
- Публичная часть сайта (отдельная фича)
