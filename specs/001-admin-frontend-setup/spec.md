# Feature Specification: Admin Frontend Setup — Landing Page & Login

**Feature Branch**: `001-admin-frontend-setup`
**Created**: 2026-03-24
**Status**: Draft
**Input**: User description: "Создать фронтенд-проект в директории front/ проекта PlovCenter. Скопировать необходимое из Farovon-LMS/Frontend для создания админки. Начальный этап — простая публичная главная страница и страница логина для админ-панели, доступная по ссылке."

## Assumptions

- Frontend проект создаётся в директории `front/` в корне репозитория PlovCenter.
- Стек берётся из Farovon-LMS: React 19 + Vite + TypeScript + MUI + TanStack React Query.
- Авторизация через существующий бэкенд PlovCenter API (`POST /api/admin/auth/login`, JWT bearer).
- Админ-панель доступна по маршруту `/admin`, публичная главная — по `/`.
- На данном этапе админ-панель содержит только заглушку (dashboard placeholder) после успешного логина.
- Публичная главная страница — минимальная: название кафе, краткое описание, ссылка на меню (заглушка).
- UI-тема создаётся своя для PlovCenter (не копируется тема Farovon).

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Посетитель открывает главную страницу (Priority: P1)

Посетитель заходит на сайт кафе и видит простую главную страницу с названием "Плов Центр", кратким описанием заведения и навигацией.

**Why this priority**: Главная страница — точка входа для всех пользователей. Без неё сайт не существует.

**Independent Test**: Открыть корневой URL `/` в браузере — отображается страница с названием кафе и базовой навигацией.

**Acceptance Scenarios**:

1. **Given** пользователь открывает `/`, **When** страница загружена, **Then** отображается название "Плов Центр" и краткое описание заведения.
2. **Given** пользователь на главной, **When** нажимает ссылку "Админ-панель", **Then** переходит на `/admin/login`.

---

### User Story 2 — Администратор входит в панель управления (Priority: P1)

Администратор переходит по ссылке `/admin/login`, вводит логин и пароль, получает доступ к админ-панели.

**Why this priority**: Логин — необходимый блок для всей будущей функциональности админки. Без авторизации ничего дальше не работает.

**Independent Test**: Ввести валидные креды (admin / Admin123!) на странице логина — редирект на `/admin` с отображением dashboard-заглушки.

**Acceptance Scenarios**:

1. **Given** администратор на `/admin/login`, **When** вводит корректные логин и пароль и нажимает "Войти", **Then** происходит редирект на `/admin` и отображается dashboard-заглушка.
2. **Given** администратор на `/admin/login`, **When** вводит неверные креды, **Then** отображается сообщение об ошибке, форма остаётся доступной.
3. **Given** администратор на `/admin/login`, **When** отправляет пустую форму, **Then** отображаются ошибки валидации на соответствующих полях.

---

### User Story 3 — Защита маршрутов админки (Priority: P2)

Неавторизованный пользователь не может попасть в админ-панель напрямую по URL.

**Why this priority**: Без защиты маршрутов логин теряет смысл. Но это вторично относительно самого UI логина.

**Independent Test**: Открыть `/admin` без токена — автоматический редирект на `/admin/login`.

**Acceptance Scenarios**:

1. **Given** пользователь не авторизован, **When** открывает `/admin`, **Then** редирект на `/admin/login`.
2. **Given** администратор авторизован, **When** открывает `/admin`, **Then** отображается dashboard-заглушка.
3. **Given** администратор авторизован, **When** нажимает "Выйти", **Then** сессия завершается, редирект на `/admin/login`.

---

### Edge Cases

- Что происходит при истечении JWT-токена? Пользователя редиректит на `/admin/login`.
- Что если бэкенд недоступен при попытке логина? Отображается понятное сообщение об ошибке сети.
- Что если пользователь вводит URL несуществующего маршрута? Отображается 404-страница.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST отображать публичную главную страницу по маршруту `/` с названием кафе и навигацией.
- **FR-002**: Система MUST отображать страницу логина по маршруту `/admin/login` с полями "Логин" и "Пароль".
- **FR-003**: Система MUST отправлять запрос авторизации на бэкенд PlovCenter API и сохранять полученный JWT-токен в клиентской сессии.
- **FR-004**: Система MUST перенаправлять авторизованного администратора на `/admin` после успешного логина.
- **FR-005**: Система MUST отображать ошибку при неверных учётных данных без перезагрузки страницы.
- **FR-006**: Система MUST перенаправлять неавторизованных пользователей с `/admin/*` на `/admin/login`.
- **FR-007**: Система MUST позволять администратору выйти из системы (logout) с очисткой сессии.
- **FR-008**: Система MUST отображать страницу 404 для несуществующих маршрутов.
- **FR-009**: Система MUST корректно работать при запуске dev-сервера с проксированием запросов к бэкенду PlovCenter.

### Key Entities

- **Session**: состояние авторизации пользователя — наличие токена, данные администратора (username).
- **LoginCredentials**: логин и пароль, вводимые на странице авторизации.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Публичная главная страница загружается и отображает контент менее чем за 2 секунды.
- **SC-002**: Администратор проходит путь от открытия `/admin/login` до попадания в dashboard менее чем за 15 секунд.
- **SC-003**: При неверных кредах ошибка отображается в течение 2 секунд после отправки формы.
- **SC-004**: Все защищённые маршруты (`/admin/*`) недоступны без авторизации — 100% редиректов на логин.
